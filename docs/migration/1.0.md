# Migrating to 1.0.0

[繁體中文](./1.0.zh-TW.md) (pending)

Helix 1.0 locks in a stable external contract. This doc tells you what changes, what stays, and what to do if you're already building on Helix 0.x today.

## TL;DR

If you're on **0.10.0** and use Helix the documented way (CLI + `helix.config.js` + SKILL.md + `/api/v1/*`), **you should have nothing to change** when 1.0 lands. No planned breaking changes at this point.

The migration work is mostly about *claiming your dependency surface*: tell us what you rely on so we keep it in the 1.0 contract.

## What 1.0 makes permanent

Once 1.0 ships, the following contracts are locked — breaking changes require a major version bump (2.0+).

### CLI commands

Stable signatures:

```
helix init
helix login [--provider P] [--api-key K]
helix start [--port N]
helix doctor
helix status
helix agent list
helix agent chat [agent_id]
helix memory stats [agent_id]
helix memory recall <agent_id> <query>
helix gateway status
helix eval run [suite]
helix eval history
helix trace runs [--limit N]
helix trace stats [--hours N]
helix trace backfill [--limit N]
helix export [--out PATH]
helix import <file>
helix workstation run "<goal>" [--timeout N] [--model M] [--poll N]
helix workstation status <task_id>
helix workstation cancel <task_id>
helix workstation health
helix workstation capabilities
helix --version
```

Additive flags may be added in minor releases. Removing flags or renaming commands would be breaking.

### HTTP API (`/api/v1/*`)

**To be frozen at 1.0.** Until 1.0 ships, current `/api/*` endpoints are still iterating. At the 1.0 cutover, the stable set gets mirrored under `/api/v1/*` (legacy `/api/*` will continue to work for at least one major version for backward compatibility).

If you use the HTTP API today, **file an issue describing which endpoints you depend on**. That feedback shapes what gets included in `/api/v1/*`.

### `helix.config.js`

Top-level keys stable from 1.0:

```js
export default {
  model:      'gemini-2.5-flash',
  apiKeyEnv:  'GEMINI_API_KEY',
  database:   { type: 'sqlite', path: '.helix/helix.db' },
  agents:     [ /* ... */ ],
  skills:     [ /* ... */ ],
  hooks:      [ /* ... */ ],
};
```

Adding new top-level keys is minor. Renaming or removing existing keys would be breaking.

### Skill contract (`SKILL.md`)

Frontmatter fields locked at 1.0:

```yaml
---
name: string
description: string
version: semver
author: string (optional)
tags: [string]
parameters:
  - name: string
    type: string|number|boolean
    required: boolean
    description: string
capabilities: [string]
---
```

The markdown body below frontmatter is free-form — no schema enforced.

### Tool registry (`register()`)

```js
registry.register({
  name:        'foo.bar',
  description: 'one-line description',
  level:       'L1' | 'L2' | 'L3',
  category:    'read' | 'write' | 'execute' | 'communicate' | 'manage',
  inputSchema: { required: string[], optional: string[] },
  handler:     async (args, context) => any,
  metadata:    {},  // optional, free-form
});
```

This shape is stable at 1.0. New levels / categories may be added; existing ones won't be removed.

### Workstation API contract

See [`docs/contracts/workstation-api.md`](../contracts/workstation-api.md). The task-spec shape (goal / success_criteria / allowed_paths / forbidden / steps / decision_policy / output_format / reset_policy / checkpoint_policy) locks at 1.0.

## What is explicitly NOT locked

These can change between minor versions (not covered by 1.0 stability promise):

- **Internal module boundaries inside `src/`.** Anything not exposed through CLI / HTTP / `helix.config.js` / `SKILL.md` / `tool-registry` is free to move. Don't `import` from `helix-agent-framework/src/...` directly — use the CLI or HTTP API.
- **Dashboard UI (`/v2/*`).** Continuously iterated; layouts and wording can change.
- **Undocumented endpoints or flags.** If it's not in the docs, don't build on it.
- **Log formats, CHANGELOG, report schemas.** Observability surface — formats may evolve.
- **Specific LLM model names.** Providers add / deprecate models on their own schedule; `helix login --provider X` API is stable, but the `model` string literal is whatever the provider exposes.

---

## Deprecations to watch

None currently planned for 1.0. If a deprecation happens between now and 1.0, it will be announced in `CHANGELOG.md` with at least one minor release of overlap before removal.

---

## Concrete upgrade checklist

If you're shipping something on top of Helix today:

- [ ] Pin your dependency: `npm install helix-agent-framework@~0.10` (allow patches, reject minor jumps) until 1.0 ships
- [ ] Check `CHANGELOG.md` on each release for anything marked **Breaking** (currently none)
- [ ] If you use the HTTP API directly: file an issue listing the endpoints you rely on
- [ ] If you use internal `src/` imports: switch to the CLI or HTTP API before 1.0
- [ ] If you wrote SKILL.md skills: confirm frontmatter uses only the documented fields (extra fields are currently ignored but may become strict at 1.0)
